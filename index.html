Introduction to Git

What is Git?

Git is a Distributed Version Control System.

Every user who clones a repository has the entire history on their machine.

Allowing you to vew the history of changes and revert to an older version easily.

Git & Github

Git is a Version Control System

Github is a hosting service for git repositories.

Github is a place to store your work (think of this as a back up, a copy of your work stored elsewhere) and collaborate with others.

Whilst Git and Github work well together, they are two separate products. Github can be replaced by any other repo hosting service. E.g. Azure, Bit Bucket etc.

What is a repository (repo)?

A repo is a collection of all of the files (and folders) you're tracking with git.

There are two sides to the repo, local and server.

Local is the copy which is on your machine and server is the copy which is on the hosting service.

Suggested Workflow

Github flow (1/2)

Github flow is a standard practice for managing a repository.

You have one main branch called master. Master contains your working code. I.e. a working SQL Script, the working website, a report which gets sent to customers. What ever it is.

You never commit changes directly to master. This way you always have the latest working copy.

Github flow (2/2)

When someone asks for changes, you create a new branch from master. This is a feature branch.

Once that feature branch is fully working, tested and signed off. You merge it in to the Master branch.

Master has now been updated to the latest version.

Basic Commands

Committing

Think of this as saving your work.

Once you have made some changes and commited them, you have a snapshot of your work at a certain point in time.

You will be able to move forward and make more changes on top of this. If you fuck up, you can roll back to this commit that you know works. Either discarding the breaking changes, or saving them to work with later.

Branching

Creating a branch creates and entirely new copy of the repo for you to work with.

You can make as many commits and changes to the new branch as you like, without affecting the original branch.

Once the work is complete, you can merge this branch in to the original one and combine the changes.

Merging

Once the work on a feature has been completed and is ready to become the main version, you merge it in to master.

This will integrate the feature branch with master.

Merge Conflicts

If master is ahead of your feature (i.e. someone else has merged to master) you will get merge conflicts which need to be sorted.

The best way to deal with conflicts is to merge master in your feature, so you have the latest changes. Fix the merge conflicts locally and then merge back to master.

Pushing

So far we have been working completely offline. The hosting service knows nothing about your changes.

When you are ready for your changes to be saved to the repo, you push the branch.

Pushing a branch syncs all of you commits with the cloud. You can work offline for days, and then sync all the changes when you get a connection.

Pulling

You pull a branch to get the latest version from the cloud on to your machine.

This is useful when multiple people are working on the same repository and you want to get their latesst changes.

Pushing & Pulling cont.

The push and pull commands work on a branch level.

You can push all of your local commits, or pull all of the server commits for a branch in one go.

You can not push or pull multiple branches in one go.

Cloning & Initializing

Cloning is pretty much what it sounds like. Take a copy of an online repository so it can be worked on from your machine. Initializing starts a new git repositiroy. It can either be an empty folder structure, or an existing one.

Installing Git

Download the installer from https://git-scm.com/

Once installed open a command prompt and run 

git --version

If the install was successful you will see the version information

Hands On (1/4)

Let's assume you already have a folder structure you would like to turn in to a git repo.

In your command prompt navigate to the folder and run the init command: 

git init .

Hands On (2/4)

Some files you don't want git to track. Passwords, secrets or nudy photos. Adding a .gitignore file will stop them being synced to the remote repo 

echo >.gitignore

Hands On (3/4)

Now you're all set up, you need to stage your files. This tells git they are ready to commit. 

git add .

Hands On (4/4)

You're ready to make your first commit. 

git commit .-m "Initial commit"

Hands On (5/5)

Sign up for a Github account and create a new repo.

Copy the commands to push an existing repository from the command line and run them from your command line. 


          git remote add origin https://github.com/JamesAskew/git-demo.git
          git push -u origin master
